# v0.14.0 — Transparency Exchange, Capability Surfacing, and Federation Comparison

This release delivers the three priorities from the Next Increment Plan: transparency publication patterns, assurance and conformance surfacing, and federation comparison workflows. Each priority produced a standalone, independently usable module with full test coverage.

## Highlights

### Transparency checkpoint publication and peer comparison (`links.checkpoint_exchange`)

- Added `sign_checkpoint(checkpoint, signing_key)` to embed an Ed25519 signature over a transparency checkpoint payload. The signature covers all fields except `signature` and `signer_key_hash` using canonical JSON serialisation.
- Added `verify_checkpoint_signature(checkpoint, verify_key)` for deterministic tamper detection.
- Added `publish_checkpoint_file(checkpoint, out_dir)` to write a timestamped checkpoint artifact to the filesystem for static HTTP serving or out-of-band peer exchange.
- Added `fetch_peer_checkpoint(base_url, village_id)` to retrieve the latest checkpoint from a remote node's `GET /villages/{village_id}/transparency/checkpoint` endpoint.
- Added `compare_checkpoints(local, peer)` which returns a `CheckpointComparisonReport` with `drift_class`, `status`, and diagnostic notes.
- Added `write_comparison_report(report, out_dir)` to persist comparison results as durable JSON artifacts under `artifacts/transparency/`.

### Drift class taxonomy (`links.drift_classes`)

- Formalised a five-class taxonomy distinguishing `aligned`, `publication_lag`, `policy_divergence`, `history_only_divergence`, and `unknown`.
- `publication_lag` and `policy_divergence` are now structurally distinct: matching policy hashes with differing entry counts is lag, not divergence. Operators no longer need to interpret raw count differences manually.
- Each drift class carries a machine-readable description, an integer severity score, and specific operator response guidance.
- Added `classify_checkpoint_drift(...)` as the primary entrypoint: given local and peer policy hashes, entry counts, and optional checkpoint hashes, returns `(drift_class, notes)`.
- Added `drift_severity(drift_class)` and `most_severe(drift_classes)` for programmatic severity aggregation across multi-village federation checks.

### Machine-readable capability declarations (`links.capability_manifest`)

- Added `build_manifest(node_id, ...)` to produce a self-describing JSON artifact declaring a node's storage mode, reconciliation mode, active transparency features, stable features, and experimental features.
- Added `verify_manifest_hash(manifest)` for integrity checks: the `manifest_hash` field is a SHA-256 digest over the canonical manifest body.
- Added `write_manifest(path, manifest)` and `load_manifest(path)` for filesystem round-trips.
- Added `check_compatibility(local, peer)` to compare two manifests and produce a structured compatibility report covering shared features, exclusive features, storage and reconciliation alignment, and federation pilot status.
- Added `schemas/capability_manifest.schema.json` (JSON Schema draft 2020-12) for interoperable validation.
- Stable and experimental feature surfaces are explicitly separated in every manifest. Experimental features signal governance maturity, not quality.

### Test coverage

59 new tests across the three modules (`test_capability_manifest.py`, `test_checkpoint_exchange.py`, `test_drift_classes.py`). All pass. Tests cover: manifest integrity, hash tamper detection, write/load round-trips, compatibility edge cases, checkpoint signing/verification idempotence, comparison classification for all five drift classes, severity ordering invariants, and operator response completeness.

## Operator acceptance criteria for federation pilots

A node is ready for a small-federation pilot when:

1. `build_manifest(node_id, federation_pilot=True, reconciliation_mode="lineage_aware")` produces a valid manifest with `manifest_hash` verifiable by peers.
2. `compare_checkpoints(local, peer).drift_class` returns `"aligned"` or `"publication_lag"` under normal operating conditions (not `"policy_divergence"`).
3. The operator has reviewed `DRIFT_CLASS_OPERATOR_RESPONSE["policy_divergence"]` and confirmed incident procedures cover that case.

## Upgrade notes

- All three modules are additive. No existing interfaces changed.
- `checkpoint_exchange` requires `pynacl` (already a project dependency) for signing; comparison and publication work without a signing key.
- `capability_manifest` and `drift_classes` have no additional dependencies beyond the standard library.
- Existing transparency checkpoint artifacts written by prior versions remain readable; signing is opt-in.
